🔑 The Meta-Insight

OpenClaw's architecture is: Gateway + Channels + Agents + Skills + Automation. GodMode is that architecture for your setup, but most of the plumbing is already built. The strategic question is: what should GodMode configure vs. rebuild vs. just expose nicely? Almost everything we've been hand-rolling in shell scripts and custom orchestration already exists in the platform — often better. The opportunity is the UX layer, not more infrastructure.

🚫 They've Already Built It — Don't Rebuild

🦞 Lobster — Typed Workflow Runtime Don't Build

This is the biggest one. Lobster is a deterministic pipeline runner with:

  • JSON-pipe chaining between steps (inbox list | inbox categorize | inbox apply)
  • Built-in approval gates — pauses, returns a resumeToken, resumes on approve
  • .lobster YAML workflow files with condition, approval, stdin: $step.stdout
  • llm-task plugin tool — structured JSON-only LLM steps inside pipelines with optional JSON Schema validation
  • Resume tokens are durable — stored state on disk, small token key handed back
  • The community "second brain" example includes: weekly-review, inbox-triage, memory-consolidation, shared-task-sync
⚠️ Current overlap: Our swarm orchestration, shell scripts, godmode-wip-commit.sh + approval patterns — all reinventing what Lobster does at the platform level.
✅ What GodMode should do: Help users build and manage .lobster workflow files through a UI. Not build another orchestration engine. Enable: tools.alsoAllow: ["lobster"]

⏰ Native Cron Scheduler Don't Build

OpenClaw has a full built-in cron system that survives restarts:

  • Persists to ~/.openclaw/cron/jobs.json
  • Two modes: main session (enqueues system event to heartbeat) or isolated session (own fresh context, announces result to channel)
  • Channel delivery: announces directly to WhatsApp/Telegram/Discord/Slack
  • Per-job model overrides (--model opus --thinking high)
  • One-shot (--at 20m), fixed interval, or cron expressions with timezone
  • Run history: ~/.openclaw/cron/runs/<jobId>.jsonl
  • Webhook delivery mode per job
⚠️ Current overlap: Our consciousness-heartbeat.sh runs via system crontab. Our morning brief, weekly maintenance — all of it could move here. The difference is dramatic: OpenClaw cron persists, delivers to channels, has run history, model overrides, restart-safe. System crontab has none of that.
✅ What GodMode should do: Expose cron job management through a UI — create/edit/disable/run/view history. This is exactly "configured but not enabled" that users need help with.

🪝 Hooks System — 4 Bundled, All Disabled Configure Now

OpenClaw ships 4 bundled hooks, disabled by default:

  • session-memory — saves session context to memory/YYYY-MM-DD-slug.md when /new is issued. Uses last 15 lines + LLM to generate filename slug (e.g., 2026-01-16-api-design.md). This is close to what our consciousness sync does but at the session boundary.
  • command-logger — JSONL audit trail of all commands (action, timestamp, session key, sender, source)
  • boot-md — runs BOOT.md at gateway startup. This is our HEARTBEAT.md startup sequence — it's already built.
  • bootstrap-extra-files — injects extra workspace files (monorepo-local AGENTS.md/TOOLS.md) during agent bootstrap

Custom hooks are TypeScript files with event listeners for: command:new, command:reset, command:stop, agent:bootstrap, gateway:startup, message:received, message:sent, and tool_result_persist (synchronously transforms tool results before session transcript).

⚠️ Current overlap: We have custom bash scripts doing what session-memory and boot-md do natively.
✅ What GodMode should do: Ship a GodMode hook pack via npm. Enable session-memory, boot-md, command-logger by default for every GodMode user. The hook pack becomes the GodMode extension point.
openclaw hooks enable session-memory
openclaw hooks enable boot-md
openclaw hooks enable command-logger

🔐 Secrets Management Configure Now

OpenClaw has a full secrets system with three provider types:

  • env — reads from environment variables
  • file — reads from a JSON file at a path
  • exec — calls any binary (1Password CLI, HashiCorp Vault, sops). 1Password integration is explicitly documented as an example.

Interactive: openclaw secrets configure wizard. Audit: openclaw secrets audit --check.

⚠️ Current overlap: Our skills/1password/SKILL.md handles 1Password retrieval for tasks. OpenClaw already handles credential storage natively — for model API keys and skill API keys. These are different use cases but worth knowing.
✅ What GodMode should do: During user setup, run openclaw secrets configure. Expose the audit as a health check in GodMode. Don't build a secrets UI — wire to the existing one.

🧠 ACP Coding Agent Sessions Don't Build

OpenClaw has a native runtime for running Codex, Claude Code, Gemini CLI, OpenCode, Pi as thread-bound sessions via ACP (Agent Client Protocol):

  • /acp spawn codex --mode persistent --thread auto
  • Thread-bound: follow-up messages in that Discord/etc thread route to the same coding session
  • /acp steer — nudges a running session without replacing context
  • Model overrides, permission profiles, timeout controls
  • Session isolation, run history, announce results back to channel
  • Harnesses: pi, claude, codex, opencode, gemini
⚠️ Current overlap: Our skills/coding-agent/SKILL.md does this manually. ACP is the designed path and works better (thread binding, steer, cancel, status, doctor).
✅ What GodMode should do: Enable ACP in config, install acpx plugin. Our coding agent skill may become unnecessary.
openclaw plugins install @openclaw/acpx
openclaw config set plugins.entries.acpx.enabled true
/acp doctor

🔀 Multi-Agent Routing Bindings Strategic

The binding system is more sophisticated than we're using:

  • Route different WhatsApp DMs to different agents by peer ID (exact phone number)
  • Route by guild, team, account, or channel-wide fallback with AND semantics
  • Per-agent sandboxing + tool restrictions (Docker containers per agent)
  • Agent-to-agent messaging (opt-in, off by default)
  • Each agent gets its own workspace, auth profiles, session store
✅ What GodMode should do: The Deck concept (agent columns) maps directly to this multi-agent model. Each GodMode "workspace" is an OpenClaw agent with its own binding config. Expose this through UI.

⚡ Memory — Far More Powerful Than We're Using

🔍 Memory Quality Settings We Haven't Enabled Configure Now

The memory system has high-value features that are all off by default:

  • Hybrid BM25 + Vector search — Not enabled. Combines semantic match (vector) with exact token match (BM25). Essential for searching code symbols, error strings, IDs — things pure vector search misses. We have 1000+ files in QMD.
  • MMR Re-ranking — Not enabled. Eliminates near-duplicate results. With months of daily notes, prevents five slightly-different versions of the same memory eating all result slots.
  • Temporal Decay — Not enabled. Exponential decay based on file age so recent memories outrank stale ones. Without this, old notes with better semantic match outrank yesterday's update on the same topic.
  • Pre-compaction memory flush — Built in. Triggers a silent agent turn before compaction to save durable notes. Uses NO_REPLY so user never sees it. Our compaction.memoryFlush.enabled: true is already on ✅
  • Memory citations — memory.citations: "auto" adds Source: <path#line> footers to memory search results.
agents: {
  defaults: {
    memorySearch: {
      query: {
        hybrid: {
          enabled: true,
          vectorWeight: 0.7,
          textWeight: 0.3,
          mmr: { enabled: true, lambda: 0.7 },
          temporalDecay: { enabled: true, halfLifeDays: 30 }
        }
      }
    }
  }
}
✅ Immediate action: Add MMR + temporal decay to openclaw.json. Hybrid is already configured but MMR and temporalDecay are missing. 3 lines, immediate quality improvement.

💓 Heartbeat — Underutilized Features

💓 Heartbeat Config We're Not Using Unused

  • Active hours — Restrict heartbeats to business hours. activeHours: { start: "08:00", end: "22:00", timezone: "America/Chicago" }. Outside window, heartbeats skip entirely.
  • Per-channel visibility controls — channels.defaults.heartbeat.showOk: false (already default), showAlerts: true, useIndicator: true. If all three are false, OpenClaw skips the heartbeat run entirely (saves API calls).
  • Per-agent heartbeats — Different agents can run on different intervals with different models. Only agents with a heartbeat block run heartbeats.
  • Model override — Already configured ✅ ("model": "anthropic/claude-sonnet-4-6")
  • includeReasoning — Sends separate Reasoning: message with heartbeat. Useful for transparency. Leave off in group chats.
✅ Recommended: Add activeHours to avoid 3am heartbeats burning tokens when you're asleep with Edison.

🎯 GodMode Strategic Opportunity

The Real Wedge

OpenClaw built a lot of infrastructure. Most of it is disabled or requires manual JSON editing to enable. Their own docs acknowledge this — they built an openclaw onboard wizard because the config is too complex. GodMode's opportunity is the UX layer on top of what they've built, not more infrastructure.

🗺️ Product Surfaces OpenClaw Doesn't Have

  • Setup wizard — Walk new users through enabling hooks, cron jobs, memory settings, secrets, channels. One screen per category. GodMode handles the JSON writes.
  • Cron job manager UI — Create/edit/disable/run/view history for scheduled tasks. This is valuable for every OpenClaw user, not just power users.
  • Workflow builder — Visual or template-based .lobster workflow builder. Build once, run forever. Approval gate UI.
  • Memory quality dashboard — Show current memory settings, enable hybrid/MMR/temporal decay with toggles, visualize search results.
  • Skills marketplace integration — ClawHub (clawhub.ai) is a public registry with vector search. GodMode could surface and install skills through UI. clawhub install <slug> from a button click.
  • Hook manager — Enable/disable/install hooks. Ship a GodMode hook pack (npm). Bundled hooks enabled by default for GodMode users.
  • Agent multi-workspace management — Each workspace gets a card. Bindings, sandboxing, tool policies — configurable through GodMode UI instead of JSON.
