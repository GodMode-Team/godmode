05 — Agent Architecture Decision

One Vicky, structured right

You asked whether to decompose Vicky into multiple agents. Here's the answer — and the architecture that makes one agent dramatically more powerful.

💡
The recommendation: Keep Vicky as one unified EA agent Peter Steinberger (creator of OpenClaw) runs his entire operation through a single agent. The reason Vicky feels unstable or hits API limits isn't because she's overloaded — it's because she lacks structured context and you're triggering her in expensive, unoptimised ways. Fix the context. Fix the model routing. Don't add agents.

❌ Multiple Siloed Agents (Don't Do)

Still hits the same API rate limits per OpenClaw install
Email agent has no calendar context → misses scheduling conflicts
Calendar agent has no CRM context → can't prioritise client meetings
More agents = more AGENTS.md files = more maintenance
You lose the "whole picture" EA effect

✅ One Vicky, Better Architecture (Do This)

Sonnet 4.6 for email triage, drafts, calendar checks (fast + cheap)
Haiku for lightweight cron jobs (e.g., conflict detection every 2hrs)
Opus for sensitive comms only (board packs, investor emails)
3 context files give her full situational awareness
Trust Tracker builds reliability over time
