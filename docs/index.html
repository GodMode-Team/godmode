How GodMode Fits Into OpenClaw

GodMode is a standalone plugin loaded by the OpenClaw gateway through the plugin system. It registers itself synchronously in register(), then defers license validation to the first RPC call. The plugin provides:

  • RPC Methods — 127 gateway methods for all GodMode features
  • Agent Tools — 5 tools (team_message, team_memory_write, coding_task, trust_rate, godmode_onboard)
  • HTTP Routes — Static file serving for GodMode UI, Deck, Mission Control proxy, and health endpoint
  • Lifecycle Hooks — 9 hooks for safety gates, team context injection, and service management
  • CLI Commands — openclaw godmode status|activate|doctor|workspace|comms|curation

State Management

GodMode uses Lit's reactive properties (@state() decorator) on the GodModeApp class. All state is defined in AppViewState (300+ properties). State mutations trigger automatic re-renders via Lit's update cycle.

Controllers

Controllers (ui/src/ui/controllers/*.ts) handle RPC calls and state updates. Each controller function takes the app state as a parameter and mutates it directly:

export async function loadMyDay(state: MyDayState) {
  state.myDayLoading = true;
  try {
    const res = await state.client.request("dailyBrief.get", { date });
    state.dailyBrief = res;
  } finally {
    state.myDayLoading = false;
  }
}

15 controller files: chat, config, my-day, sessions, workspaces, updates, trust-tracker, guardrails, focus-pulse, options, lifetracks, calendar, people, agents, mission.

Navigation

Tabs are defined in navigation.ts as TAB_GROUPS. The Tab type is a union of 24 tab names. URL routing maps paths like /godmode/chat to tabs via TAB_PATHS and PATH_TO_TAB.

Features

Onboarding (7-Phase + Wizard)

What it does: Guides new users through setting up GodMode with a conversational 7-phase flow plus an 8-step memory wizard.

How it works:

  1. Phase 0: Assessment — scans current config, produces health score (0-100)
  2. Phase 1: Identity — collects name, timezone, focus, communication style
  3. Phase 2: Second Brain — sets up memory structure and daily briefs
  4. Phase 3: Workflow Audit — maps user workflows to OpenClaw capabilities
  5. Phase 4: Configuration — audits and applies recommended settings
  6. Phase 5: First Win — live demo of a key feature
  7. Phase 6: Grand Reveal — summary of what was set up

Key files: src/methods/onboarding.ts, src/services/onboarding.ts, src/hooks/onboarding-context.ts, src/tools/onboard.ts, ui/src/ui/views/onboarding.ts, ui/src/ui/views/onboarding-wizard.ts

Daily Brief

What it does: Auto-generated daily brief with calendar, weather, tasks, carry-forward, and X Intelligence. Always-editable like Obsidian with live checkboxes that sync to tasks.

Technical flow:

  1. dailyBrief.generate fetches 6 data sources in parallel (calendar, Oura, weather, X Intelligence, carry-forward, context)
  2. Writes markdown to Obsidian vault: {VAULT}/01-Daily/{date}.md
  3. UI renders via toEditableMarkdownHtml() with enabled checkboxes
  4. Edits convert back via htmlToMarkdown(), debounced save (1500ms)
  5. Checkbox toggle flushes immediately, triggers syncTasksFromBrief()
  6. Task updates trigger syncBriefFromTasks() (no feedback loop due to fire-and-forget)

NBSP handling: Stripped at 3 layers — html-to-markdown, brief parse, brief sync.

Key files: src/methods/daily-brief.ts, src/methods/brief-generator.ts, ui/src/ui/views/daily-brief.ts, ui/src/ui/html-to-markdown.ts

Consciousness Sync

What it does: Golden sync button that runs session harvest, ClawVault sync, and heartbeat generation. Produces CONSCIOUSNESS.md in memory.

UI states: 4-state feedback (idle/loading/ok/error) with animated icon, green glow on success, red glow on error, auto-reset after 3s.

Claude Code sync: After consciousness sync, syncClaudeCodeSessions() scans ~/.claude/projects/ JSONL files and appends summaries to agent-log.

Key files: src/methods/consciousness.ts, src/services/claude-code-sync.ts

Task System

What it does: Native task management with bidirectional sync to daily brief checkboxes.

Dedup rules: Same normalized title + same date = same task. normalizeTitle() strips bold markers, em-dash context, and trailing punctuation.

Sync triggers:

  • dailyBrief.update → syncTasksFromBrief() (fire-and-forget)
  • tasks.update (status change) → syncBriefFromTasks() (fire-and-forget)
  • focusPulse.startMorningSet → syncTasksFromBrief()

Key files: src/methods/tasks.ts, src/methods/daily-brief.ts, ui/src/ui/views/workspaces.ts

Focus Pulse

What it does: Morning set workflow — extracts focus items from daily brief, tracks alignment throughout the day, calculates daily score.

Lifecycle: Start morning set → set focus → pulse checks (scored) → complete items → end day with score.

Feature toggle: focusPulse.enabled (default: true) controls topbar widget visibility.

Trust Tracker

What it does: Skill-level trust scoring where users rate agent performance (1-10) after each workflow. Tracks running average, provides feedback to improve.

Integration: After each tool call, trust-feedback.ts hook detects skill completion and queues a feedback prompt via before_prompt_build.

Guardrails (Safety Gates)

What it does: 7 configurable safety gates enforced via lifecycle hooks.

Update System

What it does: Checks npm for OpenClaw and GodMode plugin updates. "Update Now" runs openclaw update --yes with pre-update checkpoint and post-update health check.

Flow: Poll (30min) → topbar pill → click Update → checkpoint write → openclaw update → gateway restart → health check → checkpoint cleanup.

License Gate

What it does: Every RPC method is wrapped with withLicenseGate(). License key from openclaw.json plugin config.

  • Dev keys (GM-DEV-*, GM-INTERNAL) bypass remote validation
  • Production keys validated against lifeongodmode.com/api/v1/license/validate
  • 24-hour cache with grace period on network errors
  • Concurrent callers share a single validation promise

Feature Toggles

What it does: 3 toggles in the Lab tab, persisted to ~/godmode/data/godmode-options.json.

ToggleDefaultEffects
focusPulse.enabledtrueTopbar widget, morning set availability
deck.enabledfalseDeck nav link, /deck HTTP route
missionControl.enabledfalseMission Control nav item, /ops proxy

Coding Harness

What it does: Worktree-isolated coding tasks with parallel write limits, validation gates, GitHub PR creation, and iMessage notifications.

Flow: coding_task tool → complexity classification → scope inference → git worktree creation → subagent spawn → validation (lint/typecheck/test) → PR creation → optional auto-merge.

Chat

What it does: Full chat interface with streaming, attachments, session management, auto-compaction, and private mode.

  • Streaming: Block-level memoization prevents re-parsing entire message on each delta
  • Attachments: Images via gateway attachments; non-images inlined as base64 document blocks
  • Sessions: Draggable tabs with reorder, rename, close, archive
  • Auto-compact: Triggers at 90% context window usage
  • Private mode: Toggle prevents session from being saved to memory
  • Queue mode: Cmd+Enter queues messages; auto-flushes on completion

Development Guide

Setup

# Install dependencies
pnpm install

# Build everything (UI + backend + bundle)
pnpm build

# Development (UI only, with hot reload)
pnpm dev:ui

# Type check
pnpm typecheck

# Clean build artifacts
pnpm clean

Adding a New RPC Handler

  1. Create src/methods/my-feature.ts
  2. Export a Record<string, GatewayRequestHandler>: 
    import type { GatewayRequestHandler } from "openclaw/plugin-sdk";

export const myFeatureHandlers: Record<string, GatewayRequestHandler> = {
  "myFeature.get": async ({ params, respond }) => {
    // validate params, do work
    respond(true, { data: "result" });
  },
};
  3. Import and spread in index.ts: 
    import { myFeatureHandlers } from "./src/methods/my-feature.js";
// In allHandlers:
...myFeatureHandlers,
  4. Build and verify: pnpm build && pnpm typecheck

Adding a New UI View

  1. Create ui/src/ui/views/my-view.ts with render function and props type
  2. Add tab to navigation.ts TAB_GROUPS
  3. Add render case in app-render.ts
  4. Add controller in ui/src/ui/controllers/my-view.ts if needed
  5. Add styles in ui/src/styles/my-view.css and import in styles.css

Adding a Feature Toggle

  1. Add default to OPTIONS_DEFAULTS in both index.ts and src/methods/options.ts
  2. Add UI entry in ui/src/ui/views/options.ts FEATURES array
  3. Add conditional rendering in app-render.ts where the feature appears

Release Process

  1. Run pre-ship checklist: pnpm clean && pnpm build && pnpm typecheck
  2. Sync UI snapshot: pnpm ui:sync
  3. Check forbidden imports: rg "\.\./\.\./\.\./\.\./src/" -n
  4. Bump version in package.json and openclaw.plugin.json
  5. Publish: npm publish
