## 1. Anthropic Skill Creator (the meta-tool)

**Source:** `github.com/anthropics/skills/tree/main/skills/skill-creator`

Anthropic released an official Claude Code skill that helps you build, test, and evaluate other skills. This is the missing "how do you build the playbooks" piece from OpenAI's harness engineering approach.

**Why it matters for GodMode:**

The Session 2 architecture vision asked: *"How does GodMode handle new AI capabilities without code changes?"* The answer was always "files over code" — personas and skills are markdown, not TypeScript. But we never had a systematic way to CREATE those files with quality guardrails.

The skill-creator closes that loop:
- **Capture intent** → interview about edge cases → write SKILL.md → test → evaluate → iterate
- Skills with objectively verifiable outputs get automated test cases
- Quality scales through prompt engineering, measured by evals, not gut feeling
- The skill-creator itself is a skill — it's turtles all the way down

**GodMode integration:**
- Added to HARNESS.md as Section 6 ("Building Skills")
- Added `create-skill.md` to the godmode-dev workspace template
- Every GodMode coding agent now has a reference for how to build new skills properly
- This is the "how do playbooks get written" answer that was missing from the anti-fragile architecture

**Implication for the agent swarm:** Agents can now build their own skills. A researcher agent finds a recurring pattern → uses the skill-creator methodology to draft a skill → user reviews → skill gets added to the roster. The swarm teaches itself.

## 2. Google Workspace CLI (gcloud + workspace)

**What:** Google released CLI tooling for Workspace (Gmail, Calendar, Drive, Docs, Sheets). Previously, Workspace automation required OAuth web flows and REST APIs. Now it's CLI-native.

**Why it matters for GodMode:**

GodMode's "conduct, don't rebuild" principle means connecting to existing tools. Google Workspace is the biggest target:
- **Gmail** — inbox sweep, email triage, draft responses
- **Calendar** — schedule awareness, meeting prep, conflict detection
- **Drive** — file references, document sharing
- **Docs/Sheets** — content creation, data analysis

Previously this required building OAuth flows (like `gog` for Google Calendar). A CLI means GodMode agents can interact with Google Workspace the same way they interact with git — shell commands via the queue processor.

**Integration path:**
1. User installs Google Workspace CLI and authenticates
2. GodMode detects it in the environment (awareness snapshot can check for `gcloud workspace` availability)
3. Skills reference it: "Use `gcloud workspace gmail messages list` to check inbox"
4. No new TypeScript code needed — just new skill files that reference the CLI

**Priority:** Medium. Wait for the CLI to stabilize, then write skills that use it. Don't build adapters.

## Competitive Context

Both of these reinforce GodMode's anti-fragile thesis:

| New tool | Threat? | GodMode response |
|----------|---------|-----------------|
| Anthropic skill-creator | No — it's infrastructure we USE | Install it, reference it in harness, let agents build better skills |
| Google Workspace CLI | No — it's a tool we CONNECT to | Write skill files that reference CLI commands, zero new code |

Every new tool in the ecosystem makes GodMode stronger, not weaker. That's the moat.
